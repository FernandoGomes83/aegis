       Context   — 2–4 sentences of non-normative background
       User Story — "As [actor], I want [action], so that [benefit]."
       Criteria  — numbered, using SHALL / WHEN / IF vocabulary only

     Criteria vocabulary reference:
     ───────────────────────────────
       SHALL [verb]
         Use for unconditionally mandatory behavior. No exceptions.
         Example: "The system SHALL validate the signature before
         processing any payload content."

       SHALL NOT [verb]
         Use to explicitly prohibit a behavior.
         Example: "The system SHALL NOT process the request without
         first verifying the idempotency key."

       WHEN [condition], the system SHALL [behavior]
         Use for conditional triggers — a precondition that activates
         a required behavior. The WHEN clause states the trigger; the
         SHALL clause states what must happen as a result.
         Example: "WHEN a capture request is received with a
         paymentIntentId not in 'authorized' status, the system
         SHALL return HTTP 422 with error code INVALID_STATE."

       IF [condition], THEN the system SHALL [consequence]
         Use for conditional outcomes that depend on a state or
         input value rather than a triggering event.
         Example: "IF no idempotencyKey is supplied by the caller,
         THEN the system SHALL generate one derived from paymentIntentId."

     Cross-reference syntax: REQ-NNN.N

### SEC-REQ-INPUT-01: Schema Validation on All Endpoints

**Source**: `aegis/framework/security/security-requirements.yaml` — input-validation

**Priority**: Critical

**Risk**: High

**{{i18n.labels.derives_from}}**: `aegis/framework/security/security-requirements.yaml` — input-validation

**{{i18n.labels.acceptance_criteria}}**:

1. The system SHALL validate all API inputs against an explicit schema using a schema validation library (e.g., zod, pydantic, joi) before any business logic is executed.
2. Every field SHALL have a defined type, minimum and maximum length or value, and an expected format or enumeration of allowed values.
3. WHEN a request contains any field that fails schema validation, the system SHALL reject it immediately with HTTP 400 and a response body listing the specific failing fields, without exposing internal implementation details or stack traces.
4. The total request body size SHALL be limited at the middleware level to prevent denial-of-service via payload flooding.

### SEC-REQ-HEADERS-01: Security Headers on All Responses

**Source**: `aegis/framework/security/security-requirements.yaml` — headers

**Priority**: Critical

**Risk**: High

**{{i18n.labels.derives_from}}**: `aegis/framework/security/security-requirements.yaml` — headers

**{{i18n.labels.acceptance_criteria}}**:

1. The system SHALL include Strict-Transport-Security with a max-age of at least 63,072,000 seconds and includeSubDomains on every response.
2. The system SHALL include X-Frame-Options: SAMEORIGIN to prevent clickjacking on every response.
3. The system SHALL include X-Content-Type-Options: nosniff on every response.
4. The system SHALL include Referrer-Policy: origin-when-cross-origin on every response.
5. The system SHALL configure a Content-Security-Policy header beginning with `default-src 'self'`; all additional source allowances SHALL be documented and justified.
6. The system SHALL NOT use `unsafe-inline` for scripts in production; nonces or hashes SHALL be used instead.
7. CORS headers SHALL explicitly enumerate permitted origins and SHALL NOT use the wildcard Access-Control-Allow-Origin value on any endpoint that accepts authenticated requests or returns user-specific data.

### SEC-REQ-DEPS-01: Dependency Security

**Source**: `aegis/framework/security/security-requirements.yaml` — dependencies

**Priority**: High

**Risk**: High

**{{i18n.labels.derives_from}}**: `aegis/framework/security/security-requirements.yaml` — dependencies

**{{i18n.labels.acceptance_criteria}}**:

1. The dependency lockfile SHALL be committed to version control and kept up to date so that all dependency resolutions are reproducible and auditable.
2. Automated dependency vulnerability scanning SHALL be executed on every CI pipeline run.
3. WHEN a build contains a known critical or high-severity vulnerability in a direct or transitive dependency, the build SHALL fail until the vulnerability is remediated or explicitly accepted with a documented justification.
4. Critical vulnerability patches SHALL be applied within 7 days of disclosure; high-severity vulnerability patches SHALL be applied within 30 days.
5. Application secrets, database credentials, and third-party API keys SHALL be stored exclusively in environment variables or a dedicated secret manager and SHALL NOT be hard-coded in source files or committed to version control.
